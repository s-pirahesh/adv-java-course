در بخش قبل، قدرت ارث‌بری را برای ایجاد سلسله‌مراتب (IS-A) دیدیم. اکنون به یکی از قدرتمندترین و در عین حال ظریف‌ترین مفاهیم مرتبط با ارث‌بری می‌پردازیم: Casting (تبدیل نوع). Casting به ما اجازه می‌دهد که یک شیء را از طریق یک "لنز" یا "دید" متفاوت ببینیم.

تصور کنید به یک بیمارستان می‌روید. برای پذیرش، شما یک Person (شخص) هستید. اما برای پزشک، شما یک Patient (بیمار) هستید. پزشک دیدگاه خود را از "شخص" به "بیمار" تغییر می‌دهد تا بتواند کارهای تخصصی (مانند تجویز دارو) را انجام دهد. این جوهره Casting در OOP است.

۱. Upcasting: نگاه از پایین به بالا (فرزند به والد)

Upcasting یعنی گرفتن یک ارجاع از نوع فرزند و قرار دادن آن در یک متغیر از نوع والد.

  • Dog → Animal
  • Manager → Employee
  • Circle → Shape

این عمل همیشه امن و همیشه ضمنی (Implicit) است، چون هر فرزند "هست یک" والد. جاوا این کار را بدون هیچ شکایتی انجام می‌دهد.

// Dog IS-A Animal, so this is perfectly legal.
Dog myDog = new Dog("Buddy");
Animal myAnimal = myDog; // Implicit Upcasting

اثر Upcasting

وقتی شما شیء Dog را از طریق یک رفرنس Animal نگاه می‌کنید، شما به طور موقت توانایی دیدن متدهای اختصاصی Dog (مانند bark()) را از دست می‌دهید. شما فقط به متدهایی دسترسی دارید که در کلاس Animal تعریف شده‌اند.

myAnimal.eat();      // ✅ Ok, eat() is in Animal
myAnimal.sleep();    // ✅ Ok, sleep() is in Animal
// myAnimal.bark();  // ❌ Compile Error! The 'Animal' reference doesn't know what bark() is.

چرا Upcasting مفید است؟

این ویژگی، اساس چندریختی (Polymorphism) است. به ما اجازه می‌دهد تا کدی بنویسیم که با انواع مختلفی از اشیاء به صورت یکپارچه کار کند.

// این آرایه می‌تواند هر نوع حیوانی را نگه دارد!
Animal[] zoo = new Animal[3];
zoo[0] = new Dog("Rex");      // Upcast
zoo[1] = new Cat("Whiskers"); // Upcast
zoo[2] = new Lion("Simba");   // Upcast

for (Animal a : zoo) {
    a.eat(); // ما نمی‌دانیم دقیقاً چه حیوانی است، اما می‌دانیم که می‌تواند غذا بخورد.
}

۲. Downcasting: نگاه از بالا به پایین (والد به فرزند)

Downcasting عمل برعکس است: گرفتن یک ارجاع از نوع والد و تلاش برای تبدیل آن به نوع فرزند.

  • Animal → Dog
  • Employee → Manager

این عمل بالقوه خطرناک است و باید همیشه به صورت صریح (Explicit) انجام شود. چرا خطرناک است؟ چون یک Animal لزوماً یک Dog نیست؛ ممکن است یک Cat باشد!

Animal myAnimal = new Dog("Buddy"); // An Animal reference pointing to a Dog object

// Explicit Downcasting:
// "Hey Compiler, I swear this animal is a Dog. Let me treat it as one."
Dog myDog = (Dog) myAnimal;

// Now we have access to Dog-specific methods again!
myDog.bark(); // ✅ Ok!
خطر بزرگ: ClassCastException
اگر به کامپایلر دروغ بگویید، او در زمان اجرا مچ شما را می‌گیرد! 
Animal anotherAnimal = new Cat("Mitty");

// You promise the compiler this is a Dog...
Dog anotherDog = (Dog) anotherAnimal; // Compiles fine, but...

// RUNTIME ERROR! Throws java.lang.ClassCastException
// The JVM checks and says: "You lied! This object is a Cat, not a Dog."

۳. مقایسه جامع: Upcasting در مقابل Downcasting

معیار Upcasting Downcasting
تعریف تبدیل نوع از فرزند به والد. تبدیل نوع از والد به فرزند.
جهت به سمت بالا (خاص به عام). به سمت پایین (عام به خاص).
امنیت همیشه امن (Safe): هر فرزندی یک والد است. بالقوه ناامن (Unsafe): هر والدی لزوماً آن فرزند نیست.
نحوه اجرا ضمنی (Implicit): خودکار توسط کامپایلر. صریح (Explicit): باید با (Type) نوشته شود.
زمان بررسی زمان کامپایل (Compile-time). زمان اجرا (Runtime).
نتیجه خطا خطای کامپایل. ClassCastException.
دسترسی محدود می‌شود: متدهای اختصاصی فرزند دیده نمی‌شوند. گسترش می‌یابد: متدهای اختصاصی فرزند قابل دسترسی می‌شوند.
کاربرد اصلی چندریختی (Polymorphism) و کدنویسی عمومی. دسترسی به رفتارهای خاص یک زیرکلاس.

۴. نمودار حافظه Casting: تغییر رفرنس، نه شیء

یک نکته حیاتی: Casting هرگز خود شیء را در حافظه Heap تغییر نمی‌دهد. شیء همان چیزی است که با new ساخته شده. Casting فقط نوع "برچسب" یا "دید" متغیر ارجاعی (Reference) را در حافظه Stack تغییر می‌دهد.

تحلیل نمودار: هر سه رفرنس (ref1, ref2, ref3) به یک شیء واحد در Heap اشاره می‌کنند. تفاوت فقط در این است که کامپایلر به هر کدام اجازه دسترسی به چه متدهایی را می‌دهد.

۵. تور نجات: اپراتور instanceof

برای جلوگیری از ClassCastException، جاوا یک اپراتور به نام instanceof را فراهم کرده است. این اپراتور قبل از انجام Downcast، چک می‌کند که آیا شیء مورد نظر واقعاً از آن نوع هست یا نه.

public void processAnimal(Animal a) {
    // همیشه قبل از Downcast، نوع واقعی را چک کن
    if (a instanceof Dog) {
        System.out.println("Processing a Dog...");
        // چون چک کردیم، این Downcast 100% امن است
        Dog d = (Dog) a;
        d.bark();
    } else if (a instanceof Cat) {
        System.out.println("Processing a Cat...");
        Cat c = (Cat) a;
        c.meow();
    } else {
        System.out.println("Processing an unknown animal...");
        a.eat();
    }
}

۶. اشتباهات رایج و تله‌های طراحی

الف) Downcast کردن بدون چک instanceof

این رایج‌ترین اشتباه است و منجر به کدی شکننده می‌شود.

// کد شکننده
public void forceBark(Animal a) {
    // این کد فرض می‌کند که همیشه یک Dog به آن پاس داده می‌شود
    Dog d = (Dog) a; // اگر یک Cat بیاید چه؟ -> BOOM! ClassCastException
    d.bark();
}

ب) درک نکردن ClassCastException

این استثنا زمانی رخ می‌دهد که شما تلاش می‌کنید یک شیء را به نوعی تبدیل کنید که در سلسله‌مراتب IS-A آن قرار ندارد.

Object o = "Hello";
Integer i = (Integer) o; // ClassCastException: String IS-NOT-A Integer.

ج) سوءاستفاده از instanceof (Code Smell)

اگر متدی دارید که پر از if-else-if با instanceof است، این معمولاً نشانه‌ای از طراحی ضعیف است. این یعنی شما به جای استفاده از قدرت چندریختی (Polymorphism)، در حال نوشتن کد رویه‌ای (Procedural) در یک محیط شیءگرا هستید.

// طراحی بد
public void feed(Animal a) {
    if (a instanceof Dog) { /* give bones */ }
    else if (a instanceof Cat) { /* give fish */ }
    else if (a instanceof Lion) { /* give meat */ }
}

راه حل بهتر: متد eat() را در هر فرزند طوری بازنویسی (Override) کنید که غذای مناسب خودش را بخورد و سپس فقط a.eat() را صدا بزنید (در بخش چندریختی بررسی می‌شود).

۷. تمرین‌های عملی

تمرین ۱: قدرت Upcasting

  1. یک سلسله‌مراتب از کلاس‌های Publication (والد)، Book (فرزند) و Magazine (فرزند) بسازید.
  2. در main، یک آرایه از نوع Publication ایجاد کنید.
  3. یک شیء Book و یک شیء Magazine را در این آرایه قرار دهید (Upcasting).
  4. در یک حلقه for، روی آرایه حرکت کرده و یک متد مشترک (مثلاً printInfo()) را برای هر کدام صدا بزنید.

تمرین ۲: Downcasting امن

با استفاده از سلسله‌مراتب تمرین ۱، متدی بنویسید به نام getSpecificDetails(Publication p). این متد باید:

  1. چک کند که آیا p یک Book است یا Magazine.
  2. اگر Book بود، آن را Downcast کرده و متد خاص کتاب (مثلاً getAuthor()) را صدا بزند.
  3. اگر Magazine بود، آن را Downcast کرده و متد خاص مجله (مثلاً getIssueNumber()) را صدا بزند.

تمرین ۳: پیش‌بینی خطا

کد زیر در کدام خط و به چه دلیلی خطا می‌دهد؟ (خطای کامپایل یا خطای زمان اجرا؟)

class Fruit {}
class Apple extends Fruit {}
class Orange extends Fruit {}

public class Test {
    public static void main(String[] args) {
        Fruit f = new Apple();      // Line 1
        Apple a = (Apple) f;        // Line 2
        Orange o = (Orange) f;      // Line 3
        Fruit f2 = new Fruit();     // Line 4
        Apple a2 = (Apple) f2;      // Line 5
    }
}

تمرین ۴: طراحی متد چندریختی (Polymorphic)

یک متد performAction(Employee e) طراحی کنید. Employee دو فرزند Manager و Developer دارد.

  • اگر e یک Manager باشد، متد approveVacation() را صدا بزنید.
  • اگر e یک Developer باشد، متد writeCode() را صدا بزنید.
  • از instanceof و Downcasting امن استفاده کنید.

تمرین ۵: تحلیل و ریفکتور کد

کد زیر را تحلیل کنید. مشکل طراحی آن چیست؟ چگونه می‌توان آن را با استفاده بهتر از مفاهیم OOP (و بدون instanceof) بهبود داد؟

class Drawer {
    public void draw(Shape s) {
        if (s instanceof Circle) {
            System.out.println("Drawing a circle.");
        } else if (s instanceof Rectangle) {
            System.out.println("Drawing a rectangle.");
        }
    }
}

خلاصه بخش دوازدهم

  • ✅ Upcasting: تبدیل فرزند به والد. همیشه امن و ضمنی است. برای دستیابی به چندریختی استفاده می‌شود.
  • ✅ Downcasting: تبدیل والد به فرزند. بالقوه خطرناک و باید صریح باشد. برای دسترسی به رفتارهای خاص فرزند استفاده می‌شود.
  • ✅ instanceof: اپراتور امنیتی برای چک کردن نوع واقعی یک شیء قبل از Downcast.
  • ✅ ClassCastException: خطای زمان اجرا که در صورت Downcast نامعتبر رخ می‌دهد.
  • ✅ Casting: فقط "دید" یا "نوع رفرنس" را تغییر می‌دهد، نه خود شیء واقعی در حافظه.
